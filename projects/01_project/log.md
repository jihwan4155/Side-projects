### 2026-04-02 (Day 3)
- **오늘의 목표:**
  - API 수집 데이터와 SQLite 연동 및 자동 저장 로직(ETL) 구현
  - 다중 키워드 수집을 위한 `keywords` 테이블 연동 및 자동화
- **진행 상황:**
  - 기능의 모듈화: `fetch_and_save_news(keyword)` 함수를 만들어 재사용성 확보
  - 데이터 클리닝 심화: 뉴스 제목 내 불필요한 HTML 태그와 특수문자를 제거
  - 키워드 관리 자동화: 코드에 검색어를 직접 쓰지 않고, DB에서 키워드 목록을 불러와 반복문으로 전체 수집하도록 개선
  - 중복 방지: `INSERT OT IGNORE` 를 통해 이미 저장된 뉴스는 건너뛰고 새로운 뉴스만 저장하는 로직 궇녀
- **오늘의 도전 & 해결 (TIL):**
  - 문제: 뉴스 수집을 반복할 때마다 중복 데이터 처리가 까다로웠음
  - 해결: SQL의 `UNIQUE` 제약 조건과 `IGNORE` 키워드를 활용해 코드 복잡도를 낮추고 데이터 무결성 지킴
- **다음 할 일:**
  - DB에 쌓인 뉴스들을 조건에 맞춰(최신순, 안 읽은 순 등) 출력하는 조회 기능 구현
  - 특정 뉴스를 읽었을 때 `is_read` 상태를 `1`로 바꾸는 업데이트 기능 구현

### 2026-04-04 (Day 4)
- **오늘의 목표:**
  - 수집된 데이터를 관리하는 CRUD 기능 구현
  - 사용자 입력을 받아 동작하는 터미널 기반 메뉴 (CLI) 구축
  - `is_read` 상태를 활용한 유스 읽기 습관 트래킹 로직 완성
- **진행 상황:**
  - 메인 루프 구축: `while True` 문을 사용하여 사용자가 종료를 선택할 때까지 프로그램이 유지되도록 설계
  - 조회(READ) 기능: `WHERE is_read = 0` 조건을 사용해 아직 읽지 않은 뉴스만 필터링하여 보여주는 `get_unread_news()` 함수 구현
  - 수정(Update) 기능: 사용자가 입력한 특정 ID의 뉴스 상태를 '읽음'으로 변경하는 `mark_as_read()` 함수 구현
- **오늘의 도전 & 해결 (TIL):**
  - 문제: input()으로 받은 숫자로 메뉴를 선택할 때 자꾸 else 문(잘못된 입력)으로 빠지는 현상 발생.
  - 해결: input()은 데이터를 항상 문자열(str)로 받는다는 점을 확인. if choice == 1 대신 if choice == "1"로 비교하거나 int()로 형변환
- **다음 할 일:**
  - 뉴스를 읽을 때 단순히 상태만 바꾸는게 아니라, 실제 브라우저로 링크를 열어주는 기능추가
  - 수집된 뉴스 중 특정 단어가 포함된 뉴스만 골라내는 검색 기능
